# InventIP

**Inventory Intelligence Platform for Automotive Discrete Manufacturing**

InventIP is a full-stack inventory management platform that combines standard supply chain formulas, machine learning, and an agentic AI copilot to move procurement decisions from reactive to predictive. Built for ADAS component supply chains managing 15 raw materials, 3 finished products, and 7 suppliers.

---

## Features

- **Live health monitoring** — real-time status for all materials with NORMAL / REORDER / CRITICAL / STOCKOUT / OVERSTOCK classification
- **ML-powered forecasting** — Ridge Regression per material predicts next 3 months of consumption
- **Dynamic thresholds** — Safety Stock, ROP, and EOQ driven by ML forecast instead of static annual averages
- **Lead time prediction** — Linear Regression predicts actual delivery days per PO
- **90-day breach simulation** — projects when each material will stockout and calculates order-by date
- **Supplier intelligence** — weighted scoring model selects best supplier per urgency level
- **Material detail view** — per-material deep dive with consumption history, PO tracker, supplier scorecard, BOM usage
- **Stock value breakdown** — category and material-level capital analysis
- **Full PO lifecycle** — raise, confirm, receive with CSV write-back
- **AI Copilot (multi-agent)** — natural-language chat over live inventory data; a LangGraph supervisor routes each question to an Inventory, Procurement, Risk, or Policy agent
- **RAG knowledge base** — policy/SOP questions answered from company documents via FAISS retrieval, with source citations

## Installation

**1. Clone the repository**

```bash
git clone https://github.com/yourname/InventIP.git
cd InventIP
```

**2. Create a virtual environment** (recommended)

```bash
python -m venv venv

# Windows
venv\Scripts\activate

# Mac / Linux
source venv/bin/activate
```

**3. Install dependencies**

```bash
pip install -r requirements.txt
```

**4. Configure environment**

Copy the example config and fill in your local paths:

```bash
cp .env.example .env
```

Open `.env` and update the paths:

```env
BASE_DIR=C:/your/path/InventIP
DATA_DIR=C:/your/path/InventIP/dataset
MODELS_DIR=C:/your/path/InventIP/models
```

**5. Enable the AI Copilot** (optional but recommended)

Create a free API key at [console.groq.com/keys](https://console.groq.com/keys) and add it to `.env`:

```env
GROQ_API_KEY=gsk_your_key_here
```

Without the key the dashboard still works fully — only the AI Copilot page is disabled.

> **Tip:** if you use [uv](https://docs.astral.sh/uv/), skip steps 2–3 and just run
> everything with `uv run ...` — dependencies install automatically from `uv.lock`.

## Usage

**Step 1 — Train the ML models**

Run once before first use. Re-run at the start of each new month to retrain on fresh data.

```bash
python ml_model.py
```

Expected output:
```
Training Model 1: Lead Time Prediction (Linear Regression)...
  R2:  0.xxx
  MAE: x.xx days
  Saved: lead_time_model.pkl

Training Model 2: Demand Forecast (Ridge per material)...
  MAT001   R2=0.xxx  MAE=xx.x units
  ...
  Trained 15 material models.
  Saved: demand_forecast_models.pkl
```

**Step 2 — Verify dynamic values** (optional)

```bash
python app.py
```

All 15 materials should show `[DYN]` with correct SS, ROP, and avg daily values.

**Step 3 — Launch the app**

```bash
streamlit run dashboard.py
```

Opens at `http://localhost:8501`

**Step 4 — Talk to the AI Copilot**

Open the **AI Copilot** tab in the top navigation and ask questions like
*"Which materials need reordering?"*, *"Which supplier should I use for Bearing
Set?"* or *"What are the PO approval limits?"*. Each answer shows which agent
handled it and which tools it called.

You can also chat from the terminal:

```bash
uv run python -m agents.cli "What could stock out in the next two weeks?"
```

After editing the documents in `knowledge_base/`, rebuild the RAG index:

```bash
uv run python -m rag.build_index
```

## ML Models

### Model 1 — Lead Time Prediction

Predicts actual lead time in days from PO date to goods received. Used directly as `predicted_lt` — no adjustment to nominal lead time needed.

| | |
|---|---|
| **Algorithm** | Linear Regression + StandardScaler |
| **Training data** | Delivered POs with known actual delivery date |
| **Target** | `actual_lead_days` = actual_delivery_date − po_date |
| **Features** | supplier_encoded, quantity_ordered, month_of_year, supplier_rating, order_value, promised_days |
| **Output** | Predicted actual lead time in days |
| **Saved as** | `models/lead_time_model.pkl` — `{model, scaler, le}` |

### Model 2 — Demand Forecast

15 Ridge Regression models — one per material. Predicts next 3 months of consumption. Powers dynamic avg_daily used in all ML-driven threshold calculations.

| | |
|---|---|
| **Algorithm** | Ridge Regression (alpha=1.0) + StandardScaler |
| **Training data** | Monthly consumption from daily sales × BOM |
| **Target** | `total_consumed` — total units consumed per material per month |
| **Features** | month_index, month_of_year, lag_1, lag_2, lag_3, rolling_3m_avg |
| **Evaluation** | Leave-one-out cross-validation (12 predictions per material) |
| **Saved as** | `models/demand_forecast_models.pkl` — `{models, scalers, metrics, last_features}` |

`last_features` contains precomputed lag values per material saved at training time. This allows instant prediction at runtime without re-scanning the monthly DataFrame — reduced Planning tab load time from 30 seconds to under 0.5 seconds.

## AI Copilot Architecture

```
question ──> Supervisor (intent router, LangGraph)
               ├── Inventory Agent    stock, statuses, safety stock, ROP, EOQ
               ├── Procurement Agent  reorders, supplier choice, POs, forecasts
               ├── Risk Agent         stockout projections, order-by dates
               └── Policy Agent       RAG over knowledge_base/ documents
                        │
               8 LangChain tools ──> app.py / ml_model.py outputs (zero new math)
```

- **LLM** — Groq API, `llama-3.3-70b-versatile` (free tier). One model, four
  "personalities": each agent gets its own system prompt and tool subset.
- **Tools, not prompts** — agents never see raw CSVs; they call typed tools that
  wrap the same `app.py` functions the dashboard uses, so chat answers always
  match the numbers on screen.
- **RAG pipeline** — `knowledge_base/*.md` → section chunking → fastembed
  embeddings (`bge-small-en-v1.5`, local ONNX) → FAISS similarity search →
  passages cited in the answer. The policy documents mirror the exact constants
  in the code (1.65 z-score, 40/30/30 supplier scoring, ABC 70/90 cutoffs).
- **Routing rule** — questions mixing live data with policy go to a data agent
  (they can also search policies); the Policy agent handles pure rule questions.

## Core Formulas

```
EOQ           = sqrt(2 × D × S / H)
Safety Stock  = 1.65 × (avg_daily × 0.25) × sqrt(lead_time)
Reorder Point = avg_daily × lead_time + Safety Stock
Days Remaining = current_stock / avg_daily
```

**Static** (Operational Dashboard) — uses `annual_demand / 365` for avg_daily and nominal lead time from supplier map. Always available without ML.

**Dynamic** (Planning & Forecast) — uses `(M+1 + M+2 + M+3) / 3 / 30` for avg_daily from Model 2 and `model.predict()` for lead time from Model 1.

## Notes

- `models/*.pkl` are excluded from version control — regenerate by running `python ml_model.py`
- `.env` is excluded from version control — copy `.env.example` and configure locally (paths + `GROQ_API_KEY`)
- Current month is excluded from ML training data — forecasts remain stable when daily sales are entered mid-month
- Models retrain when the month changes — run `python ml_model.py` at the start of each month
- The RAG index in `rag/index/` is generated — rebuild with `python -m rag.build_index` after editing `knowledge_base/`
- The embedding model (~130 MB) downloads automatically the first time the RAG index is built
